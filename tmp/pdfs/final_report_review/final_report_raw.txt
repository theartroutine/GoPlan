1.1 Project Background
Grouptripsarerarelyplannedbyonepersonfrombeginningtoend. Friends, fam-
ilies, and teams usually discuss destinations in chat, compare possible dates, invite par-
ticipants, create an itinerary, divide costs, and keep important memories after the trip. In
practice, these activities are often scattered across messaging applications, spreadsheets,
note-taking tools, payment screenshots, and photo albums. The result is duplicated in-
formation, missing context, unclear responsibility, and difficulty checking the final cost
of the trip.
GoPlan is designed as a web application that centralizes those coordination tasks
into a shared trip workspace. The system supports account onboarding, friend connec-
tions, trip creation, member invitation, itinerary planning, expense tracking, settlement,
notifications,realtimechat,AI-assistedplanning,photomanagement,andmemoryvideo
generation. The project is implemented as a full-stack application using Next.js, Django
REST Framework, PostgreSQL, Redis, WebSocket, and an AI provider integration.
This final report documents the completed system as a software engineering cap-
stone project. It presents the final scope, implemented architecture, design decisions,
business modules, testing approach, limitations, and future development directions.
1.2 Problem Statement
The main problem addressed by this project is the lack of a single collaborative
workspaceforgrouptravelplanning. Agroupneedstoanswerseveralquestionsthrough-
out the lifecycle of a trip:
• Who is participating in the trip, and who has permission to manage important
information?
• What is the itinerary for each day, including time, location, activity status, as-
signee, and reminders?
• Which expenses have been recorded, who should share each cost, who has con-
tributed money, and what transfers are needed for settlement?
1• How can trip members discuss plans in realtime while keeping the conversation
connected to the trip data?
• How can AI assist planning without directly changing critical trip data without
user confirmation?
• How can members preserve trip photos and generate shareable memory videos
after or during the trip?
Existing general-purpose tools can solve parts of these questions, but they do not
provide a consistent data model, permission model, realtime flow, and trip-specific user
experience. GoPlan addresses this by combining structured trip management with real-
time collaboration and AI-assisted action drafting.
1.3 Objectives
The main objective of the project is to design and implement an AI-driven group
trip planning and coordination system. The specific objectives are:
• Build a maintainable full-stack system with clear boundaries between the browser
UI,Next.jsBFFroutehandlers,DjangoRESTAPI,servicelayer,andPostgreSQL
database.
• Implement secure account and session management, including registration, login,
logout, email verification, profile setup, password reset, password change, avatar
update, access-token refresh, and current-user loading.
• Supportsocialcoordinationthroughfriendsearch,friendrequests,friendlists,trip
invitations, trip membership, and role-based trip management.
• Providetripplanningfeaturesincludingtripmetadata, destinations, coverimages,
itinerary sections, activities, custom activity types, activity statuses, assignees,
locations, and reminders.

2.1 Group Trip Planning Systems
A group trip planning system must combine social coordination, schedule man-
agement, expense accounting, and communication. Unlike a personal travel planner, the
system must handle multiple users acting on shared data. This introduces business rules
for invitations, member roles, visibility, edit permission, concurrency, auditability, and
notification delivery.
GoPlan treats a trip as the central workspace. Most business objects belong to
a trip: members, invitations, timeline sections, activities, expenses, settlements, chat
messages,photos,andmemoryvideos. Thismakesthepermissionmodeleasiertoreason
about because access can be checked through active trip membership.
2.2 Backend-for-Frontend Architecture
The project uses a Backend-for-Frontend architecture, abbreviated as BFF. A BFF
is a server-side layer created specifically for a frontend application. In this project, the
browsercallsNext.jsRouteHandlersfirst,andthoseroutehandlerscalltheDjangoREST
API.
Browser → Next.js Route Handlers → Django REST API → PostgreSQL / Redis
This design is important for authentication and API consistency. The browser
does not call Django directly for HTTP APIs. Route handlers can forward access tokens,
manage refresh-token cookies, normalize response shapes, proxy protected assets, and
hide server-only API keys such as the HERE location search key.
2.3 REST API and Service Layer
The backend uses Django REST Framework for HTTP APIs. Views are kept thin:
they parse request data, call serializers for validation when needed, delegate business
6logic to services, and return structured responses. The service layer contains the main
business rules such as trip creation, invitation acceptance, timeline mutation, expense
settlement, chat message sending, AI draft execution, and memory video creation.
This separation reduces duplication and makes backend tests more focused. Busi-
ness behavior can be tested at the service level, while API tests verify authentication,
permission, throttling, and response contracts.
2.4 JWT Session Model
The system uses JSON Web Tokens, abbreviated as JWT, for authenticated API
access. AJWTisasignedtokenthatallowsthebackendtoverifyauser’sidentitywithout
storing a traditional server-side session for every request. In GoPlan, the access token is
short-lived and the refresh token is longer-lived.
The security rule is that the refresh token is stored only in an HttpOnly cookie. An
HttpOnly cookie cannot be read by browser-side JavaScript, which reduces the damage
of cross-site scripting attacks. The access token is kept in memory on the frontend and
is not stored in localStorage or sessionStorage. When the access token expires, the BFF
refresh route uses the refresh cookie to obtain a new access token.
2.5 Realtime Communication with WebSocket
The project uses WebSocket over ASGI for realtime features. WebSocket allows
a persistent connection between the browser and backend so that the server can push
notifications and chat events without requiring the browser to continuously poll.
The current realtime topology uses a direct browser-to-Django WebSocket con-
nectionat/ws/realtime. ThisdoesnotchangetheHTTP/BFFrulebecausenormalHTTP
API calls still go through Next.js Route Handlers. WebSocket authentication is handled
with short-lived WebSocket tickets issued through the BFF. The ticket is used during the
WebSocket handshake, and the backend validates the ticket through ASGI middleware
before accepting the connection.
The realtime consumer is multiplexed, meaning notification events and trip chat
subscriptions share one WebSocket connection. This reduces connection overhead and
centralizes session validation.
Page 7 / 1142.6 AI-Assisted Planning
The AI feature is integrated into trip chat. A user can mention GoPlanAI in a trip
conversation and request help with planning actions such as creating a timeline activity,
updating an expense, or finalizing settlement. The AI provider returns tool calls, and the
backend converts those tool calls into action drafts.
The action-draft design is a safety boundary. The AI does not directly mutate
important trip data. Instead, it creates a draft with a summary, display payload, missing
fieldsifany, requiredconfirmationtype, andexpirytime. Auserwiththecorrectpermis-
sion can then confirm, edit, cancel, or complete missing information. Only confirmed
drafts are executed by backend service logic.
2.7 Media Processing and Memory Video Rendering
The project includes media features for trip covers, photos, and memory videos.
Photo uploads are validated by file type, size, pixel count, decoded memory budget, and
allowed image formats. The system generates derivative assets such as thumbnails and
medium-size images to improve frontend performance and reduce unnecessary transfer
size.
Memory videos are rendered from trip photos. The backend stores video render
state, selected photos, source mode, music key, generated video file, poster file, and
public sharing information. Long-running render work is queued through Celery and
Redis so the HTTP request can return quickly while the render process continues in the
background.

3.1 Actors
The system includes the following main actors:
• Guest: A visitor who has not logged in. This actor can register, log in, verify
email, and request password reset.
• AuthenticatedUser: Auserwhohasloggedinandcompletedtherequiredprofile
flow. This actor can manage profile data, friends, notifications, and trip partici-
pation.
• Trip Member: An authenticated user who is an active member of a trip. This
actor can view the trip workspace, participate in chat, manage allowed activities,
add expenses, upload photos, and create memories.
• Trip Captain: A trip member with elevated trip management permission. This
actor can edit trip information, invite members, remove members, control trip
status, and confirm captain-managed AI drafts.
• AI Assistant: A system actor represented in chat messages. It reads trip context
and creates action drafts through backend-controlled tools.
• ExternalServices: Emailserviceforverification/resetemail,HERElocationser-
vice for location search, DeepSeek AI provider for AI completions, Redis for
channel layer and task queue, and FFmpeg/Pillow-based media processing tools
for image/video work.

4.1 Architecture Overview
GoPlan is implemented as a layered full-stack system. The browser renders the
user interface with Next.js and React. HTTP requests are sent to Next.js Route Handlers,
which form the BFF layer. The BFF then calls the Django REST API. Django services
handlebusinesslogicandpersistdatainPostgreSQL.RedissupportsWebSocketchannel
delivery and Celery task queues.
Next.js UI → BFF Route Handlers → DRF API Views → Services → Models / PostgreSQL
For realtime communication, the browser opens a WebSocket connection to
Django ASGI. Before opening the connection, the frontend obtains a short-lived Web-
Socket ticket through a BFF route. The WebSocket middleware validates that ticket and
attaches the authenticated user to the connection scope.
4.2 Frontend Design
The frontend uses Next.js App Router. User-facing pages are organized into
authentication routes, shell routes, trip workspace routes, and public memory-sharing
routes. Important pages include login, registration, profile setup, dashboard, friends,
trip creation, trip overview, timeline, expenses, members, chat, photos, memories, and
public memory share view.
Feature code is grouped under frontend/features. Each feature generally sepa-
rates domain types, infrastructure API clients, application hooks/state, and presentation
components. This structure helps keep UI components focused on rendering and inter-
action while API calls and domain transformations remain reusable.
The BFF layer is implemented under frontend/app/api. The repository currently
contains route handlers for authentication, friends, invitations, notifications, realtime
tickets, trips, timeline, expenses, settlement, chat, AI drafts, photos, memories, public
memory assets, media upload, and location search.
214.3 Backend Design
The backend is a Django project using Django REST Framework and ASGI. The
main local apps are:
• accounts: custom user model, authentication, email verification, profile setup,
avatar, password flows, throttling, and JWT authentication.
• friends: friend request lifecycle, friendship persistence, identify-tag search, val-
idators, permissions, and services.
• trips: trip model, trip members, invitations, trip status transitions, timeline sec-
tions, custom types, activities, reminders, and location data.
• expenses: expenses, participants, contributions, settlement, transfers, and ledger
entries.
• notifications: notificationpersistence,unreadcounts,mark-as-readbehavior,and
service-triggered realtime events.
• realtime: WebSocket tickets, middleware, routing, and the multiplexed realtime
consumer.
• chat: trip chat messages, reactions, hidden messages, deletion behavior, AI men-
tion parsing, and WebSocket push helpers.
• ai: AI interactions, DeepSeek provider integration, tool-calling agent, context
building, action drafts, draft execution, and background tasks.
• media: trip cover upload, public media serving, and image validation helpers.
• memories: trip photos, photo variants, memory video rendering, music catalog,
private/public assets, and share links.
4.4 Authentication and Authorization Design
Authentication uses a custom user model with email as the login identifier. User
profile completion is separate from account creation so the system can guide users
through email verification and identify-tag setup before allowing full collaboration
features.
Page 22 / 114Authorization is primarily based on active trip membership and role. Many
trip-specific operations first verify that the actor is an active member of the target trip.
Captain-only operations require the actor to have the CAPTAIN role. This keeps the
permission model consistent across trips, timeline, expenses, chat, photos, memories,
and AI action drafts.
Session security uses short-lived access tokens and rotating refresh tokens. The
refresh token is stored in an HttpOnly cookie through the BFF. The access token is
kept in memory and forwarded by BFF routes to Django. The system also stores an
auth_version on the user model so already-open WebSocket sessions can be invalidated
when the user’s authentication state changes.

Page 31 / 1144.7 Realtime Design
Realtime behavior is implemented through Django Channels and Redis chan-
nel layers. The browser connects to /ws/realtime with the expected subprotocol.
The connection is authenticated by a short-lived WebSocket ticket issued through
/api/realtime/ws-ticket.
After connection, the consumer joins a user-specific notification group. For chat,
the client sends a chat.subscribe message with a trip identifier. The consumer checks
that the current user can access that trip chat before joining the trip chat group. This
prevents users from subscribing to trips where they are not active members.
The same WebSocket connection can receive notification events, chat messages,
message deletion events, and chat membership kick events. The consumer also checks
whether an open WebSocket session remains valid before processing messages or for-
warding events.
4.8 AI Action Draft Design
The AI feature is built around two persistent models: AIInteraction and AIAc-
tionDraft. An AIInteraction records one AI request with provider, model, status, token
usage, latency, errors, and task information. An AIActionDraft records a proposed ac-
tion generated by the AI.
Supported AI draft action types include:
• Timeline activity create, update, delete, and status update.
• Expense create, update, delete, and contribution update.
• Settlement finalization, reopening, transfer marked sent, and transfer confirmed
received.
Each action type has a required confirmation category. Captain-managed actions
requiretripcaptainconfirmation. Activitystatusupdatesrequireconfirmationfromarel-
evant activity actor. Transfer actions require payer or recipient confirmation depending
on the operation. This design allows the AI to assist planning while preserving explicit
user control over data mutations.

Page 44 / 114Chapter 5. Implementation
5.1 Authentication and Profile Module
The authentication module supports registration, login, logout, refresh, email ver-
ification, password reset, password change, profile setup, profile name update, avatar
update, and current-user loading. The backend stores users in a custom accounts.User
model with email as the username field and profile fields such as first name, last name,
display name, identify name, identify code, email verification status, profile completion
status, and avatar.
The identify tag combines identify_name and identify_code. This allows users
to search for each other without exposing internal database identifiers. The profile com-
pletion constraint ensures that a completed profile contains all required fields.
On the frontend, authentication screens are implemented under the (auth) route
group. The auth context manages current-user state and access-token memory. BFF auth
routes handle login, refresh, logout, registration, verification, password reset, and profile
setup. The refresh token remains in an HttpOnly cookie managed by the BFF.
5.2 Friends Module
The friends module allows users to build a social graph before planning trips to-
gether. The backend stores pending requests in FriendRequest and accepted relation-
shipsinFriendship. Servicesenforcerequestlifecyclerulessuchaspreventingduplicate
active requests, preventing self-requests, and converting accepted requests into friend-
ships.
The frontend provides screens for friend list, adding friends, and viewing incom-
ing/outgoing requests. BFF routes proxy friend operations to Django so the browser
continues to follow the HTTP architecture rule.
5.3 Trip Management Module
The trip module is the center of the application. A trip contains metadata such as
name, destination, date range, description, currency, timezone, budget estimate, cover
45image, status, and destination provider information. The system supports PLANNING,
ONGOING, COMPLETED, and CANCELLED statuses.
Trip membership is stored in TripMember, with roles CAPTAIN and MEM-
BER. A trip starts with the creator as an active captain. Invitations are stored in Trip-
Invitation, and accepting an invitation activates membership through backend services.
Captains can invite friends, remove members, edit trip information, and control trip life-
cycle transitions.
The frontend trip workspace is organized into tabs such as overview, timeline,
expenses, members, chat, photos, and memories. This creates a focused interface where
members can move between planning areas without leaving the trip context.
5.4 Timeline Module
The timeline module represents itinerary planning. TimelineSection stores trip
days or date-based sections. TimelineActivity stores individual activities inside a sec-
tion. Activities support different time modes such as all-day, flexible, exact time, and
time range. They also support statuses, assignee scope, structured or manual location
data, notes, meeting points, contact details, booking references, and external links.
The module also supports custom activity types through TimelineCustomType.
This lets each trip define its own activity categories while still supporting predefined
systemtypessuchastransportation,accommodation,food,sightseeing,shopping,check-
in/check-out, free time, and other.
Timeline reminders are represented by TimelineActivityReminder. Each re-
minder stores an offset and a UTC due time so reminder dispatch can be performed
consistently in the backend.
5.5 Expense and Settlement Module
The expense module supports collaborative cost tracking. An Expense stores ti-
tle, description, total amount, currency, collector, creator, and locking state. Participant
shares are stored in ExpenseParticipant. Paid or contributed amounts are stored in Ex-
penseContribution.
The settlement flow calculates balances from current expenses and contributions.
A finalized settlement is represented by TripSettlement, and individual transfer instruc-
Page 46 / 114tions are represented by SettlementTransfer. The model enforces one finalized settle-
ment per trip. A settlement can be reopened when the trip’s financial data needs further
changes.
The module also records important money-related events in ExpenseLedgerEn-
try. Ledger entries provide a trace of expense creation, updates, deletion, contribution
changes, settlement finalization, reopening, and transfer status changes.
5.6 Notification Module
The notification module stores user notifications in PostgreSQL and supports un-
read counts and mark-as-read behavior. Notification types include friend requests, ac-
cepted friends, trip invitations, invitation responses, trip cancellation, member removal,
and trip timeline reminders.
When a notification is created, the backend can push it through the realtime chan-
nel layer to the connected user’s WebSocket group. The frontend notification context
and notification bell display unread state and notification items.
5.7 Realtime Chat Module
Trip chat is implemented as a combination of HTTP APIs and WebSocket push.
HTTP APIs provide message history, gap fill, updated-since queries, message creation,
deletion,hiding,andreactions. WebSocketdeliversnewlycreatedmessagesanddeletion
events to subscribed trip members.
TheChatMessagemodelstoresthetrip, sender, sendersnapshots, senderkind, AI
status,content,clientmessageid,timestamps,anddeletionmetadata. Theclientmessage
id provides idempotency for send retries. MessageReaction stores one reaction per user
per message, and ChatMessageHiddenForUser stores per-user hidden messages.
The chat service validates content length, active trip membership, trip mutabil-
ity, deletion windows, reaction emoji validity, and hidden-message visibility. When a
message mentions GoPlanAI, the service creates an AI interaction and enqueues AI pro-
cessing after the database transaction commits.
Page 47 / 1145.8 AI Assistance Module
The AI module connects chat to backend-controlled planning actions. The agent
builds a trip context bundle containing members, timeline data, expenses, settlement
state, recent chat, active drafts, and target versions for stale-data checks. It sends this
context and the user’s prompt to the DeepSeek provider using tool-calling.
The backend defines tool schemas for creating, updating, deleting, and status-
changing timeline activities; creating, updating, deleting, and setting contributions for
expenses; finalizing or reopening settlement; marking transfers sent; confirming trans-
fers received; updating existing drafts; and responding with a plain text message.
When the provider returns tool calls, the backend validates tool arguments using
schemas before invoking handlers. Handlers create or update AIActionDraft records
rather than directly mutating trip data. Confirming a draft later executes the correspond-
ing backend service operation. This protects the system from accidental AI changes and
gives users the final decision.
5.9 Media and Photos Module
The media module supports trip cover uploads and public media file serving. The
memories module extends media handling with trip photos. Each trip photo stores up-
loader snapshots, original dimensions, a thumbnail asset, a medium asset, and times-
tamps.
Photo upload validation includes file count, per-file size, total upload size, source
dimensions, decoded byte budget, image header detection, and allowed web image for-
mats. This is important because image files are user-controlled input and can consume
excessive memory if processed without limits.
The frontend provides photo grid, lightbox, upload review, asset proxying, and
download flows. Protected photo assets are retrieved through authenticated BFF routes
so private trip photos are not exposed as unrestricted public files.
5.10 Memories Module
The memories module allows trip members to create memory videos from trip
photos. A memory video stores creator snapshots, title, status, source mode, selected
Page 48 / 114photo ids, photo count, music key, generated video file, poster file, duration, render error
fields, render recovery count, Celery task id, and optional public share information.
Thesystemsupportsmanualphotoselectionandautomaticphotoselection. Mem-
ory video rendering is queued through Celery and processed in the background. Render
status can be queued, rendering, ready, or failed. The service enforces limits such as
minimum/maximum photos and maximum active render jobs per user or trip.
Ready memory videos can be downloaded privately by trip members. A ready
memory can also enable a public share slug, allowing a separate public memory page to
display the video and poster without requiring trip membership.
5.11 Location Search Integration
Location search is implemented through frontend BFF route handlers that call the
HERE API server-side. This keeps the HERE API key out of browser code. Timeline
activities and trip destinations can store structured provider information such as provider
id, title, address, latitude, longitude, and country code.
5.12 Local Infrastructure
The project uses Podman Compose for local infrastructure. The backend uses
PostgreSQL for relational persistence, Redis for Channels and Celery, Mailpit for de-
velopment email, and Daphne/ASGI for WebSocket-capable serving. The frontend runs
separately through Next.js development or build commands.

Page 111 / 114Chapter 8. Conclusion and Future Development
8.1 Conclusion
This project implemented GoPlan, an AI-driven group trip planning and coordi-
nation web application. The system centralizes the main needs of group travel planning:
account onboarding, friend relationships, trip workspace management, itinerary plan-
ning, expense settlement, notification delivery, realtime chat, AI-assisted actions, photo
handling, and memory video sharing.
From a software engineering perspective, the project demonstrates a maintain-
ablefull-stackarchitectureusingNext.js,DjangoRESTFramework,PostgreSQL,Redis,
WebSocket, Celery, and AI provider integration. The system applies clear layer bound-
aries, service-oriented backend logic, BFF-based HTTP integration, security-focused to-
ken handling, trip-based authorization, and focused automated tests.
The most important design decision in the final system is the separation between
AI suggestion and data mutation. GoPlanAI can interpret trip context and create action
drafts, but it cannot silently change core trip data. Users must confirm drafts through the
application, and backend services execute only approved actions.
8.2 Limitations
The current project still has limitations:
• The UML diagrams in this report are reused from an earlier stage and do not yet
fully cover chat, AI, media, photos, and memories.
• Full end-to-end testing across the complete user journey is not yet documented in
this report.
• Productiondeployment hardening, monitoring, backup, and operational runbooks
are not fully covered.
• The task coordination module is not included in the implemented final scope.
• AI quality depends on provider availability, prompt quality, context construction,
and action schema coverage.
112• Memory video rendering depends on background worker resources and media
processing performance.
8.3 Future Development
Future development can focus on:
• Redrawing UML diagrams to reflect the final implemented system, especially AI,
chat, media, photos, and memories.
• Adding full end-to-end tests for the main demo journey.
• Implementing task coordination for checklist-style trip responsibilities.
• Improving deployment automation, observability, logging, and production secu-
rity configuration.
• ExpandingAIassistancewithsaferitinerarysuggestions, budgetanalysis, context
summarization, and multi-step planning support.
• Adding richer notification coverage for expense changes, memory render com-
pletion, chat mentions, and AI draft events.
• Improving media storage scalability by integrating object storage for production
deployments.
